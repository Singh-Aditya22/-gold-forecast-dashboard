# Gold Forecast Dashboard — Claude Context

Read this first when opening Claude Code in this repo (e.g. on a different machine after
cloning) — it has the context needed to keep working on this project without re-deriving
everything from scratch.

## What this is

An end-to-end personal DE + ML project: tracks non-physical gold investment options in
India (Gold Futures, Gold ETFs, SBI Gold Fund), forecasts prices with 7 models, and helps
identify historical/predicted buying dips. See `README.md` for the full pitch, stack, and
design-decision writeup — this file is the "how to keep developing it" companion.

**This is the canonical, actively-developed repo.** It was originally seeded inside a
larger `de-portfolio` archive repo (`09_gold_forecast_dashboard/` there); that copy is now
a frozen historical snapshot and should not be edited — all real work happens here.

## Multi-machine development

This project is worked on from more than one machine (personal Ubuntu laptop + office
laptop), synced via this GitHub repo (`Singh-Aditya22/-gold-forecast-dashboard`, public).
There's no other coordination mechanism — just `git pull` before starting, `git push` when
done, same as any shared repo. A couple of things are **machine-specific and don't travel
with the code**:

- **systemd `--user` services** (auto-start dashboard on login, daily catch-up refresh
  timer, auto-publish to GitHub) are set up on the personal laptop only
  (`~/.config/systemd/user/gold-forecast-{dashboard,refresh}.service` + `.timer`). On any
  other machine, just run things manually (`streamlit run dashboard/app.py`,
  `./refresh.sh`) unless you want to replicate that setup there too.
- **`.streamlit/config.toml`** binds the dashboard to `127.0.0.1` only (localhost-only,
  no LAN/WiFi access) — this was a deliberate choice on the personal laptop; reconsider
  per-machine if that's not the right default elsewhere.
- The personal laptop's `refresh.sh` auto-commits and pushes `gold_forecast.duckdb` after
  each successful daily refresh (see `refresh.sh`'s step 7). If you also run refresh
  locally elsewhere, you'll get the same behavior — be aware two machines running it
  independently could both try to push data, first one wins, second one needs a pull first.

## Architecture

```
pipeline/     collect.py → bronze.py → silver.py → gold.py     (data pipeline, DuckDB medallion)
models/       train.py → evaluate.py → predict.py → track_predictions.py → backfill_live_predictions.py
dashboard/    app.py (Streamlit UI) + charts.py + queries.py + insights.py
refresh.sh    runs the full pipeline + auto-publishes to GitHub
```

- **7 models per instrument**, ALL generated by `predict.py` (not just the winner): naive
  (random-walk baseline — the domain-standard accuracy bar, not an arbitrary MAPE%),
  Prophet, ARIMA (both with macro exogenous regressors: VIX/oil/USD index/10Y yield),
  XGBoost, LightGBM (hyperparameter-tuned), LSTM (PyTorch), ensemble (average of the
  other 6, not retrained separately).
- **Return-space modeling**: XGBoost/LightGBM/LSTM predict next-day *log-return*, not raw
  price level, then reconstruct price by compounding — training on price level directly
  means a tree model can't extrapolate past its training-range max, which silently
  produces terrible forecasts once the market moves into new territory. This one change
  flipped an earlier "naive wins everywhere" result to naive genuinely losing to LightGBM
  on 2 of 3 instruments.
- **Damped-trend compounding** (`DAMPING_PHI=0.90` in `models/common.py`): a recursive
  multi-step forecast can turn a small daily bias into a wildly unrealistic long-horizon
  move (a steady +0.4%/day compounds to +45% over 90 days). Predicted returns are damped
  toward zero as the horizon lengthens.
- **Live forward-testing** (`models/track_predictions.py`): separate from `evaluate.py`'s
  one-time historical backtest (which evaluates each model *frozen* as of Dec 2024
  against all of 2025 — a much harder, longer-horizon task) — this logs every model's
  real next-day prediction daily and reconciles it once the actual price arrives. Because
  the backtest and the live task are different difficulty profiles, don't be surprised if
  live rankings diverge from the backtest's "best choice" — that's expected, not a bug;
  worth revisiting the backtest methodology (e.g. rolling walk-forward) if live data keeps
  disagreeing with it over time.
- **Gap backfill** (`models/backfill_live_predictions.py`): if the laptop is asleep/off on
  a given day, no prediction ever gets logged for it — this closes that gap by retraining
  each model using ONLY data strictly before the missed day (genuine walk-forward, no
  lookahead) and immediately reconciling since the outcome is already known. No depth
  limit by design (a long gap means real per-model retrain time, LSTM the slowest at ~1
  min each — there's no honest shortcut). Scoped to the window since live tracking
  actually started, not the full multi-decade history. Runs as refresh.sh step 7/8.
- **Outlier-guarded ingestion**: a rolling-median check in the silver layer catches vendor
  data glitches before they corrupt technical indicators or training data (a real one hit:
  GOLDBEES.NS briefly printed ~1% of its actual price for two days in Dec 2019).

## Two requirements files

- `requirements.txt` — full set, needed for the pipeline/model scripts (prophet, xgboost,
  lightgbm, pmdarima, torch, yfinance, mftool). Install torch separately with the CPU-only
  index (`pip install torch --index-url https://download.pytorch.org/whl/cpu`) to avoid
  pulling CUDA wheels.
- `requirements-dashboard.txt` — just `duckdb`/`pandas`/`numpy`/`streamlit`/`plotly`, since
  `dashboard/app.py` and everything it imports only reads the pre-built
  `gold_forecast.duckdb`, never touches the training libraries. Used for the Streamlit
  Community Cloud deploy (fast, light build).

## Deployment

Live on Streamlit Community Cloud (auto-redeploys on every push to `main`, reading
`requirements-dashboard.txt`). See README for the deploy steps if setting this up fresh
on a new Streamlit Cloud account.

## Known simplifications (already accepted, not bugs to "fix" reflexively)

- Future macro regressor values are carried forward flat (not themselves forecast) over
  the 90-day horizon.
- The evaluate.py backtest gives Prophet/ARIMA "perfect foresight" on 2025's actual macro
  values, which real deployment can't have (see [[project_qbo_copy_into_plan]]-style
  caveats aren't relevant here, but the same kind of backtest-vs-live gap applies).
- `hdfc_gold_etf` has the shortest history (since June 2023) — expect noisier
  small-sample behavior for that instrument specifically.
